## 1\. 개요

### 1.1 문서 목적

본 문서는 MES(Manufacturing Execution System) 및 APS(Advanced Planning & Scheduling) 시스템의 PI(Process Innovation) 활동을 효과적으로 지원하기 위한 **'PI Wiki 시스템'의 기능적/비기능적 요구사항**을 정의하는 것을 목적으로 한다.

본 문서는 개발 착수 전 이해관계자 간 합의를 도출하고, 개발 범위와 기준을 명확히 하기 위해 작성되었다.

### 1.2 프로젝트 배경

- 현재 MES/APS 레거시 시스템에 대한 소스코드 분석을 통해 프로세스 분석, 문제점, 개선사항이 다수 도출되고 있음  
- 도출된 분석 산출물이 개인 PC, 메일, 메신저 등에 분산되어 있어 체계적 관리가 어려움  
- PI 참여 시스템 담당자와 현업 담당자 간 의사소통 및 의사결정 이력 추적의 어려움  
- 즉흥적으로 도출되는 아이디어, Pending된 의사결정 항목 등을 체계적으로 관리할 도구의 부재  
- 기존 문서 도구(Word, Excel)는 검색·버전관리·협업 측면에서 한계 존재

### 1.3 시스템 명칭 및 정의

| 구분 | 내용 |
| :---- | :---- |
| **시스템 명칭** | PI Wiki (가칭) |
| **시스템 정의** | MES/APS PI 활동에서 발생하는 모든 분석 산출물, 의사결정, 아이디어를 체계적으로 관리하고 협업할 수 있는 위키 기반 지식 허브 |
| **핵심 가치** | 체계적 관리 · 빠른 검색 · 원활한 협업 · 의사결정 추적 |

### 1.4 적용 범위

- MES/APS PI 프로젝트 전 기간 (분석 → 개선안 도출 → TO-BE 설계)  
- PI TF 시스템 담당자 및 현업 담당자 (약 30명)  
- 사내 인트라넷 환경에서의 운영

### 1.5 용어 정의

| 용어 | 정의 |
| :---- | :---- |
| **PI** | Process Innovation. 업무 프로세스 혁신 활동 |
| **MES** | Manufacturing Execution System. 제조 실행 시스템 |
| **APS** | Advanced Planning & Scheduling. 고도화 계획·스케줄링 시스템 |
| **AS-IS** | 현행 모델 (현재의 프로세스/시스템 상태) |
| **TO-BE** | 목표 모델 (개선 후 도달하고자 하는 상태) |
| **문서(Page)** | PI Wiki에서 작성·관리되는 단위 콘텐츠 |
| **스페이스(Space)** | 최상위 카테고리. 예: 현행분석, 개선방안 등 |
| **아이디어 보드** | 포스트잇 형태로 즉흥적 아이디어를 등록·논의하는 칸반 보드 |

## 2\. 시스템 개요

### 2.1 시스템 개요도

본 시스템은 다음과 같은 4개의 주요 영역으로 구성된다.

- **계층형 메뉴(트리) 영역**: 모든 콘텐츠를 PI 방법론 또는 사용자 정의 구조로 탐색  
- **문서(Wiki) 영역**: 마크다운/리치텍스트 기반 문서 작성·열람·편집  
- **협업 영역**: 코멘트, 멘션, 토론, 의사결정 로그  
- **아이디어 보드 영역**: 즉흥적 아이디어 및 Pending 항목 칸반 관리

### 2.2 주요 사용자

| 사용자 그룹 | 규모 | 주요 활동 |
| :---- | :---- | :---- |
| **시스템 담당자** | 약 10명 | 소스코드 분석 결과 작성, 문제점·개선안 정리, 기술적 의사결정 주도 |
| **현업 담당자** | 약 15명 | 현행 업무 프로세스 검증, 개선안 검토 및 의견 제시, 의사결정 참여 |
| **PM/리더** | 약 5명 | 전체 진행 상황 모니터링, Pending 이슈 결정, 우선순위 조정 |

### 2.3 기대 효과

- PI 산출물의 중앙 집중화 및 단일 진실 공급원(Single Source of Truth) 확보  
- 문서 검색 시간 단축 및 분석 결과 재활용성 향상  
- 의사결정 이력의 명확한 추적으로 거버넌스 강화  
- 비대면 협업 활성화 및 현업-시스템팀 간 커뮤니케이션 효율화  
- PI 종료 후에도 영구적 지식 자산으로 활용 가능

## 3\. 기능 요구사항

기능 요구사항은 우선순위에 따라 다음 3단계로 구분한다.

- 🔴 **필수(Must)**: 1차 오픈 시점에 반드시 포함되어야 하는 기능  
- 🟡 **중요(Should)**: 1차 오픈에 포함되거나 직후 빠르게 제공되어야 하는 기능  
- 🔵 **선택(Could)**: 여건이 허락할 때 제공되는 부가 기능

### 3.12 화이트보드 (브레인스토밍 캔버스) — v1.2 신규

**칸반 보드 vs 화이트보드 사용 가이드**

1. **PI 킥오프 브레인스토밍**: "현재 MES의 모든 문제점을 일단 다 적어보자" → 화이트보드에 자유 추가 → 그룹화 → 투표로 우선순위 → 상위 항목을 페이지로 변환  
2. **개선 아이디어 발산**: "TO-BE 시스템에 어떤 기능이 들어가야 할까?" → 화이트보드에 떠오르는 대로 → 카테고리별 그룹 → 정식 요구사항 페이지로 승격  
3. **회의 중 시각적 정리**: 회의 중 실시간으로 의견을 화이트보드에 정리 → 회의 종료 시 회의록 페이지로 변환

## 11\. 가정 및 제약사항

### 가정사항

- 사내 인트라넷 환경에서 모든 사용자가 동등한 접근 권한을 가진다 (1차)  
- Docker가 설치된 사내 서버 자원이 사전에 확보된다  
- 사내 표준 브라우저는 Chrome 또는 Edge 최신 2개 버전이다  
- 외부망 접근이 가능하여 npm/Docker Hub 이미지 다운로드가 가능하다

### 제약사항

- 1차 오픈 시점에는 사용자 인증 기능이 포함되지 않는다  
- 동시 편집(co-editing) 기능은 제공하지 않는다 (편집보다 열람 중심, **단 Edit Lock으로 충돌 방지**)  
- 첨부파일 본문 내용 검색은 1차 범위에서 제외한다 (파일명 검색만 지원)  
- 벡터 검색을 활용한 RAG 기능은 1차 범위에서 제외한다 (인프라만 준비)

## 부록 C. v1.2 변경 사항 (화이트보드 추가)

### 추가 배경

Confluence Whiteboards 기능을 참조하여, **시각적 브레인스토밍** 도구를 별도로 추가. 기존 칸반 아이디어 보드(FR-601\~608)와 보완 관계로 사용한다.

### 칸반 vs 화이트보드 사용 분리

| 도구 | 사용 시점 | 특징 |
| :---- | :---- | :---- |
| **칸반 아이디어 보드** | 수렴 단계 (Pending 추적) | 정해진 컬럼, 상태 흐름, 구조화 |
| **화이트보드** | 발산 단계 (브레인스토밍) | 무한 캔버스, 자유 위치, 시각적 그룹핑 |

### 신규 기능 요약 (FR-1201\~1213)

| 영역 | 기능 |
| :---- | :---- |
| **캔버스** | 무한 캔버스, 줌/팬, 점 격자 배경, 미니맵 |
| **요소** | 포스트잇(7색), 그룹 박스(Frame), 화살표, 도형, 텍스트, 이모지 스티커 |
| **상호작용** | 자유 드래그/리사이즈, 스냅 가이드, 투표(👏) |
| **변환** | 화이트보드 → 정식 페이지 자동 변환 (그룹 박스 → H2, 포스트잇 → 불릿) |
| **관리** | 트리에서 페이지와 동일하게 관리, 코멘트, PNG 내보내기 |

### 기술 스택 추가

| 영역 | 선정 기술 | 사유 |
| :---- | :---- | :---- |
| 캔버스 라이브러리 | **tldraw** | 오픈소스, React 친화적, 무한 캔버스/줌/팬/스냅 모두 기본 제공. 자유도 높은 커스터마이징 가능 |
| 캔버스 데이터 저장 | JSON (PostgreSQL JSONB) | tldraw의 snapshot 형식을 그대로 저장. 스키마 변경 없이 유연 |

`Whiteboard`, `WhiteboardElement`, `WhiteboardVote`, `WhiteboardConversion`
